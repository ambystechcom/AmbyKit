# AGENTS.md — AmbyKit

Guidance for AI coding assistants working **on the AmbyKit codebase itself**. (For using AmbyKit
in your own project, see `README.md` and `docs/`.)

## What AmbyKit is

AmbyKit is a **Spec-Driven Development (SDD)** framework distributed as a Node/TypeScript CLI
(`@ambystech/ambykit`, bin `ambykit`). It helps AI coding assistants build better software by
producing structured artifacts — a project **constitution**, **user stories**, testable
**requirements**, **UI design**, an implementation **plan**, and an ordered **task** list — before
code is written.

The core design is **author-once, emit-per-tool**: one neutral source of truth (templates + phase
prompts) is translated by per-tool **emitters** into each assistant's native command/rules format
(Claude Code, OpenCode, GitHub Copilot + CLI, Cursor + CLI, Antigravity IDE + CLI).

## Golden rules

1. **Author once, emit per tool.** Never hand-maintain per-tool command files as source. The neutral
   source lives in `src/templates/` and `src/prompts/`; emitters in `src/emitters/` generate tool
   files (e.g. `.claude/commands/amby.*.md`, `.github/prompts/`). If you find yourself editing those
   by hand, stop — edit the neutral source and re-run `ambykit sync` instead.
2. **`BaseEmitter` / `BaseCommand` hold shared logic.** New tool support = a thin subclass of
   `BaseEmitter` that sets its `commandSurface`/paths and overrides only what differs. New CLI verb =
   a subclass of `BaseCommand` implementing `execute()`. Don't duplicate orchestration.
3. **Token frugality is a feature.** Phase prompts and templates must minimize tokens the assistant
   reads/writes: progressive disclosure (load only the artifact a phase needs via `@path`), reference
   by stable ID instead of restating, and patch-in-place rather than regenerate. See
   `docs/architecture.md` → Token efficiency.
4. **We dogfood AmbyKit on AmbyKit.** Features are specced under `specs/` using the AmbyKit workflow.
   The repo's own `.claude/` etc. are generated by `ambykit sync` and must stay in sync (CI checks).

## Project structure

```
src/cli/         BaseCommand + add/analyze/check/dashboard/init/restore/sync/update + registry
src/core/        CommandSpec model, template loader, feature-dir/numbering, dependency graph
src/emitters/    BaseEmitter + per-tool subclasses + registry (target→emitter map)
src/templates/   neutral artifact templates (constitution, spec, ui, design-tokens, plan, tasks)
src/prompts/     neutral phase CommandSpecs (constitution → implement)
.claude/commands/amby.*.md   generated Claude Code commands (from src/prompts/) — do not hand-edit
test/            emitter snapshot tests + CLI e2e
docs/            user + contributor documentation
.amby/           AmbyKit's OWN constitution/config (dogfooding)
specs/           AmbyKit's OWN feature specs (dogfooding)
```

## Build & test

- `npm run build` — compile TypeScript to `dist/` (ESM, NodeNext).
- `npm run typecheck` — type-check without emitting.
- `npm test` — run Vitest (emitter snapshot tests + CLI e2e).
- Node ≥ 20, ESM only (`"type": "module"`). Use `.js` extensions in relative imports.

## Emitter contract (read before touching `src/emitters/`)

Each emitter maps a neutral `CommandSpec` + rules context to native files. Get the exact per-tool
paths and frontmatter keys from `docs/tool-compatibility.md` — they differ in ways that bite:
- Rules file: `CLAUDE.md` (Claude, needs `@AGENTS.md` bridge) vs native `AGENTS.md` (everyone else).
- Command surface: `commands` (`.prompt.md`/`.md`) vs `skills` (`SKILL.md`, Copilot CLI) vs
  `workflows` (`.agents/workflows/*.md`, Antigravity) vs `none` (Cursor CLI, rules-only).
- Scoped instructions: Copilot `applyTo` vs Cursor `globs`.
- MCP top-level key: `.mcp.json`/`servers`/`mcpServers`/`mcp` — and scope (project vs user-level).

When adding/altering an emitter, update `docs/tool-compatibility.md` and add a snapshot test.

## Conventions

- Emitted slash-command namespace is `amby.*` (e.g. `/amby.specify`).
- Consumer feature dirs are numbered `specs/NNN-slug/`.
- Stable IDs: user stories `US-#`, requirements `FR-###`, success criteria `SC-###`, tasks `T###`.
- Prefer small, pure functions in `src/core`; keep side effects (fs, process) in `cli`/emitter edges.

## Don't

- Don't fabricate tool config formats — verify against `docs/tool-compatibility.md` (sourced from
  official docs). If a format is uncertain, flag it, don't guess.
- Don't write user-level files (`~/.copilot/mcp-config.json`, etc.) without explicit consent; keep
  them out of the "repo in sync" check.

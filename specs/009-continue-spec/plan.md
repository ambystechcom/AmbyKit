# Implementation Plan — Continue / edit an existing feature's spec & design (`/amby.revise`)

> The HOW. Satisfies every `FR-###` in `@spec.md` and honors `@../../.amby/constitution.md`.
> References spec by ID — does not restate it. No `ui.md` (this feature ships a workflow prompt, not UI).

## Technical context

- **Stack:** existing AmbyKit source-of-truth prompts (`src/prompts/*.md`) + the emit pipeline; Node
  ≥ 20, TS (ESM), Vitest. Astro Starlight docs site.
- **Key libraries:** none new. The command is a **neutral prompt file** loaded by
  `loadCommandSpecs` (`src/core/command-spec.ts`) and emitted per tool by `buildEmittedFiles` — the
  existing author-once / emit-per-tool machinery (Principle 1).
- **Constraints driving choices:**
  - Principle 1 — author `src/prompts/revise.md` once; every tool file is generated by `ambykit sync`.
  - Principle 3 — the prompt instructs *patch in place, preserve IDs, add next-number items* — no
    regeneration; mirrors how `clarify` edits only what it must.
  - Principle 2 — no duplicated logic: reuse the CommandSpec/emitter path; the dashboard is unchanged.
  - Principle 6 — after adding the prompt, `ambykit sync` and the docs-sync check must stay green
    (a new site workflow page is required — see below).

## Architecture

**This feature is almost entirely a prompt.** The behavior (continue, done-guard, ID preservation,
status transition, design continuation) is authored as instructions the coding agent executes — exactly
like `specify`/`clarify` — not as new TypeScript. Concretely:

```
  src/prompts/revise.md            ← NEW. The whole /amby.revise behavior (US-1/2/4).
     frontmatter: id: revise, name: amby.revise, phase: revise,
                  reads: [spec.md, ui.md], writes: [spec.md, ui.md], allowedTools: [read, edit]
     body: 1. Resolve the target feature dir; if absent → error, no changes (FR-007).
           2. Read spec.md frontmatter status. IF done → make no edits, tell the author to
              run /amby.specify for a new feature (FR-003, US-2). Distinct from clarify/specify (FR-008).
           3. Else continue: add/refine US-#/FR-###/SC-### IN PLACE, preserving every existing ID
              and text; new items take the next available number (FR-001/002, Principle 3).
              Leave any done stories untouched; new stories may be added (FR-004).
           4. Set spec.md status: in-progress (FR-004).
           5. Design: if ui.md exists, continue it in place; else direct to /amby.design (FR-009, US-4).

  src/core/rules.ts                ← add "revise" to PHASE_SEQUENCE (workflow-table ordering only).

  (generated by `ambykit sync`)    ← .claude/commands/amby.revise.md, .github/prompts/…, etc. (FR-006)

  site/src/content/docs/workflow/revise.mdx   ← NEW page (docs-sync A1: one page per CommandSpec id).
```

**No `src/*.ts` logic** beyond the one-line `PHASE_SEQUENCE` addition. The done-guard, ID rules, and
status transition are enforced by the prompt, not code (see Risks).

### FR-005 — dashboard needs no change (Phase 0 finding)

`computeDashboard` derives **story-level** status directly from each story's `**status:**` line
(`src/core/dashboard.ts:97-98`); the status vocabulary already includes `in-progress`, and the
006 `status-badge` tokens already color it. So once `/amby.revise` sets statuses correctly, the
dashboard reflects in-progress vs done with **zero code change** — FR-005 is a verification + a test,
not a feature. (Evidence in `research.md` R-1.)

## Phased approach

- **Phase 0 — Research** → `research.md`: (R-1) confirm the dashboard already reflects status →
  no code change; (R-2) where `revise` sits in `PHASE_SEQUENCE` and the site sidebar order (docs-sync
  A3 contiguity).
- **Phase 1 — Foundation:** create `src/prompts/revise.md` (frontmatter + skeleton) and add `revise`
  to `PHASE_SEQUENCE`; it now loads + emits. No data-model/contracts (nothing new to model).
- **Phase 2 — US-1:** author the continue/preserve-IDs/next-number body.
- **Phase 3 — US-2:** author the done-guard + redirect + status→in-progress + distinctness from
  clarify/specify.
- **Phase 4 — US-3:** verification test that a continued (in-progress) story renders in-progress on the
  dashboard, and a done feature stays done.
- **Phase 5 — US-4:** author the design-continuation branch (edit `ui.md` in place / redirect to design).
- **Docs & self-host:** add the site workflow page + `docs/workflow.md` mention; run `ambykit sync`;
  keep docs-sync + emitter tests green.

## Requirement mapping

| Requirement | How it's satisfied |
|---|---|
| FR-001 | `revise.md` body: continue spec in place, no regeneration |
| FR-002 | Prompt rule: preserve all existing IDs/text; new items take the next number |
| FR-003 | Prompt done-guard: `spec.md` status `done` → no edits, redirect to `specify` |
| FR-004 | Prompt: set status `in-progress`; leave done stories untouched, allow new stories |
| FR-005 | No change — `dashboard.ts` already parses `**status:**` incl. `in-progress` (R-1); test only |
| FR-006 | New CommandSpec auto-emits to every tool via `buildEmittedFiles` (Principle 1) |
| FR-007 | Prompt: feature-not-found → clear error, no changes |
| FR-008 | Prompt scope explicitly distinct from `clarify` (markers) and `specify` (new feature) |
| FR-009 | Prompt design branch: continue `ui.md` in place, or redirect to `/amby.design` if absent |
| US-1..US-4 | Phases 2–5; each independently testable per its spec block |

## Risks & decisions

- **Behavior is prompt-enforced, not code-enforced.** ID preservation, the done-guard, and the status
  transition depend on prompt quality — there's no TypeScript gate. Deterministic tests cover the
  *mechanics* (prompt loads, valid frontmatter, emits to every tool, docs-sync green); the *agent
  behavior* is validated by a scripted scenario/dogfood, not a unit test. This matches how
  `specify`/`clarify` are already verified. Accepted; noted so reviewers don't expect code guards.
- **Docs-sync will fail until the site page exists** (A1: one workflow page per CommandSpec id; A3:
  contiguous `order`). Adding `revise` requires `site/src/content/docs/workflow/revise.mdx` with the
  right `phase`/`command`/`reads`/`writes` and an `order` that keeps 1..N contiguous. Decision: insert
  `revise` after `clarify` and renumber the following pages (mechanical), so the sidebar reads in
  workflow order. (R-2)
- **Emitter/e2e tests that count emitted files** may need `+1`; snapshot the new command file. Minor.
- **`reads`/`writes` include `ui.md`** (design continuation, FR-009); harmless for spec-only features
  since the prompt only touches `ui.md` when it exists.
